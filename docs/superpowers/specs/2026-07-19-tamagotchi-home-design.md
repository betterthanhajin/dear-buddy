# 다마고치 홈 컴포넌트 설계

## 목표

검수되지 않은 기존 MVP 화면을 `/backup`으로 옮기고, `/`에서는 작은 다마고치형 홈 컴포넌트 하나만 보여줍니다. 첫 단계에서는 화면 구조와 한 가지 돌봄 반응에 집중합니다.

## 참고 이미지

- `dear-buddy-reference/tv-frame.png`: 세로형 핑크 TV 프레임과 물리 버튼 3개
- `dear-buddy-reference/game-scenes.png`: 흰 배경의 픽셀 펫, 상단 HUD, 하단 상태 바, 돌봄 장면의 미술 방향

`tv-frame.png`는 앱의 기기 외장으로 사용합니다. `game-scenes.png`는 화면 구성과 픽셀 표현의 참고 자료로만 사용하며 잘라서 최종 자산으로 쓰지 않습니다.

## 화면 구성

`TamagotchiDevice`는 모바일 우선 단일 컴포넌트입니다.

1. 기기 프레임은 원본의 2:3 비율을 유지합니다.
2. 화면 영역에는 레벨, 애정 수치, 픽셀 펫, 상태 바 3개를 배치합니다.
3. 픽셀 펫은 참고 이미지와 같은 흰색 애착 인형 계열로 새로 생성한 투명 PNG를 사용합니다.
4. 프레임의 중앙 하트 버튼 위에는 접근 가능한 투명 버튼 영역을 겹칩니다.
5. 중앙 버튼을 누르면 애정 수치가 1 오르고, 펫이 짧게 뛰며 하트 반응을 보여줍니다.
6. 좌우 버튼은 첫 단계에서 비활성 상태로 두고 접근성 설명을 제공합니다.

화면 바깥에는 제목, 설명 카드, 보조 메뉴를 추가하지 않습니다. 첫 화면의 중심은 기기 하나입니다.

## 라우트와 컴포넌트

- `app/backup/page.tsx`: 기존 `DearBuddyApp`을 그대로 렌더링합니다.
- `app/page.tsx`: 새 `TamagotchiDevice`만 렌더링합니다.
- `components/TamagotchiDevice.tsx`: 애정 수치와 반응 상태를 관리하는 클라이언트 컴포넌트입니다.
- `lib/tamagotchi.ts`: 초기 상태와 애정 수치 증가 규칙처럼 화면에서 분리할 수 있는 순수 로직을 둡니다.

기존 `DearBuddyApp`, 저장 데이터, API 라우트는 수정하지 않습니다. `/backup`과 새 `/` 사이에 상태도 공유하지 않습니다.

## 자산

- `public/tamagotchi/tv-frame.png`: 제공된 TV 프레임을 복사해 사용합니다.
- `public/tamagotchi/idle-pet.png`: 참고 이미지에 맞춰 생성한 픽셀 펫 PNG입니다.

이미지 생성은 개발 도구에서 한 번만 수행하며 앱 실행 중 외부 API를 호출하지 않습니다.

## 오류 처리

이미지가 늦게 로드되더라도 기기 비율과 화면 영역은 유지합니다. 인터랙션은 브라우저 상태만 사용하므로 네트워크 오류 상태는 만들지 않습니다. 애정 수치는 최대 99로 제한합니다.

## 테스트

1. 순수 로직 테스트로 애정 수치가 1씩 증가하고 99를 넘지 않는지 확인합니다.
2. 라우트 파일이 새 컴포넌트와 기존 앱을 올바르게 분리하는지 정적 테스트로 확인합니다.
3. 전체 테스트, 린트, 프로덕션 빌드를 실행합니다.
4. 390 x 844와 데스크톱 화면에서 프레임 비율, 이미지 잘림, 버튼 위치, 가로 스크롤을 브라우저로 확인합니다.

## 완료 조건

- `/`에는 새 다마고치 기기만 표시됩니다.
- `/backup`에서는 기존 앱이 이전과 동일하게 열립니다.
- 중앙 하트 버튼을 누르면 애정 수치와 반응이 바뀝니다.
- 기존 미커밋 파일 4개에는 변경을 추가하지 않습니다.
- 자동 테스트, 린트, 빌드와 시각 검수를 통과합니다.
